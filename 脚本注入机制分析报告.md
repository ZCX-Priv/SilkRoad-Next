# SilkRoad 脚本注入机制详细分析报告

## 1. 概述

SilkRoad 代理服务器实现了一套完整的自定义 JavaScript 脚本注入系统，允许管理员在代理转发的 HTML 页面中自动插入自定义 JS 代码。该系统由三个核心模块协同工作：

- **`modules/scripts.py`** — 脚本管理器，负责加载、缓存和分发脚本
- **`modules/rewriter.py`** — URL 重写器，在链接重写过程中执行脚本注入
- **`databases/scripts.json`** — 脚本配置文件，定义脚本的注入位置和排除规则

---

## 2. 整体架构与调用链

脚本注入的完整调用链如下：

```
用户请求 → SilkRoadHTTPRequestHandler.do_request()
         → Proxy(handler, url_rewriter).proxy()
         → Proxy.process_response(r)
         → [判断 Content-Type 为 text/html]
         → url_rewriter.revision_link(r.content, encoding)
         → [在链接重写完成后，执行脚本注入]
         → script_manager.get_scripts_for_position(position, domain)
         → 将脚本内容包装为 <script> 标签插入 HTML
```

### 2.1 入口：`Proxy.process_response()`

在 [SilkRoad.py](file:///c:/Users/赵晨旭/Desktop/SilkRoad-alpha/SilkRoad.py) 的 `process_response` 方法中，当响应的 `Content-Type` 包含 `text/html` 时，会调用 `url_rewriter.revision_link()` 对 HTML 内容进行处理：

```python
# SilkRoad.py 第 762-772 行
if "text/html" in content_type:
    try:
        encoding = r.encoding or 'utf-8'
        content = self.url_rewriter.revision_link(r.content, encoding)
    except Exception:
        content = self.url_rewriter.revision_link(r.content, 'utf-8')
    
    content_length = len(content)
    self.handler.send_header('Content-Length', content_length)
    self.handler.end_headers()
    self.handler.wfile.write(content)
```

**关键点**：脚本注入仅在 HTML 页面中执行，对于 CSS、JS、图片、流媒体等其他类型的响应不会注入脚本。

## 3. ScriptManager 脚本管理器

### 3.1 类结构与初始化

`ScriptManager` 类定义在 [modules/scripts.py](file:///c:/Users/赵晨旭/Desktop/SilkRoad-alpha/modules/scripts.py) 中，是脚本注入系统的核心组件。

```python
class ScriptManager:
    def __init__(self):
        self.base_dir = os.path.dirname(os.path.dirname(os.path.abspath(__file__)))
        self.scripts_dir = os.path.join(self.base_dir, "scripts")     # 脚本文件目录
        self.scripts_cache = {}                                        # 脚本内容缓存
        self.last_load_time = 0                                        # 上次加载时间
        self.cache_ttl = 60                                            # 缓存有效期（秒）
        self.scripts_config = {}                                       # 脚本配置数据
        self.positions = {                                             # 注入位置映射
            "after_head_start": "<head>",
            "before_head_end": "</head>",
            "after_body_start": "<body>",
            "before_body_end": "</body>"
        }
```

### 3.2 四个注入位置

系统定义了四个 HTML 注入位置，每个位置对应一个 HTML 标签：

| 位置名称 | 对应标签 | 注入方式 | 实际效果 |
|---------|---------|---------|---------|
| `after_head_start` | `<head>` | 在标签**之后**插入 | 脚本出现在 `<head>` 紧后面 |
| `before_head_end` | `</head>` | 在标签**之前**插入 | 脚本出现在 `</head>` 紧前面 |
| `after_body_start` | `<body>` | 在标签**之后**插入 | 脚本出现在 `<body>` 紧后面 |
| `before_body_end` | `</body>` | 在标签**之前**插入 | 脚本出现在 `</body>` 紧前面 |

### 3.3 脚本加载与缓存机制

`get_all_scripts()` 方法负责从磁盘加载所有 `.js` 脚本文件，并实现了一个基于时间的缓存机制：

```python
def get_all_scripts(self):
    current_time = time.time()
    
    # 缓存未过期，直接返回
    if current_time - self.last_load_time < self.cache_ttl and self.scripts_cache:
        return self.scripts_cache
    
    # 重新加载所有脚本
    scripts = {}
    script_files = glob.glob(os.path.join(self.scripts_dir, "*.js"))
    
    for script_path in script_files:
        script_name = os.path.basename(script_path)
        with open(script_path, 'r', encoding='utf-8') as f:
            script_content = f.read()
        scripts[script_name] = script_content
    
    # 更新缓存
    self.scripts_cache = scripts
    self.last_load_time = current_time
    return scripts
```

**缓存策略**：
- 缓存有效期为 60 秒（`cache_ttl = 60`）
- 过期后重新从磁盘读取所有 `.js` 文件
- 这意味着修改脚本文件后，最多需要 60 秒才能生效

### 3.4 按位置获取脚本

`get_scripts_for_position()` 是脚本注入的关键方法，它根据位置和域名过滤脚本：

```python
def get_scripts_for_position(self, position, domain=None):
    scripts = self.get_all_scripts()
    position_scripts = []
    
    for script_config in self.scripts_config:
        script_name = script_config.get("name")
        script_position = script_config.get("position", "before_body_end")
        exclude_domains = script_config.get("exclude_domains", [])
        enabled = script_config.get("enabled", True)
        
        # 检查：位置匹配 + 文件存在 + 已启用
        if script_position == position and script_name in scripts and enabled:
            # 检查域名排除
            if domain and any(domain.endswith(excluded) for excluded in exclude_domains):
                continue
            position_scripts.append((script_name, scripts[script_name]))
    
    return position_scripts
```

**过滤逻辑**（必须同时满足）：
1. 脚本配置的 `position` 与当前注入位置匹配
2. 脚本文件实际存在于 `scripts/` 目录中
3. 脚本的 `enabled` 字段为 `True`
4. 当前请求的域名不在脚本的 `exclude_domains` 列表中

### 3.5 脚本启用/禁用管理

`update_script_status()` 方法允许动态更新脚本的启用状态：

```python
def update_script_status(self, script_name, enabled):
    for script_config in self.scripts_config:
        if script_config.get("name") == script_name:
            script_config["enabled"] = enabled
            break
    else:
        return False
    
    # 持久化保存到 scripts.json
    config_data = {
        "scripts": self.scripts_config,
        "positions": self.positions
    }
    with open(config_path, 'w', encoding='utf-8') as f:
        json.dump(config_data, f, ensure_ascii=False, indent=2)
    return True
```

## 4. 脚本注入的核心实现

### 4.1 注入位置：`rewriter.py` 的 `revision_link()` 方法

脚本注入的实际执行发生在 [modules/rewriter.py](file:///c:/Users/赵晨旭/Desktop/SilkRoad-alpha/modules/rewriter.py) 的 `revision_link()` 方法中（第 619-641 行），在完成所有 URL 重写操作之后：

```python
# rewriter.py 第 619-641 行
# 在不同位置插入自定义JS脚本
domain = parse.urlparse(self.url).netloc

# 检查是否启用脚本插入功能
if config.get("SCRIPT_INJECTION_ENABLED", True):
    # 处理所有定义的位置
    for position, tag in script_manager.positions.items():
        if tag in content_str:
            position_scripts = script_manager.get_scripts_for_position(position, domain)
            if position_scripts:
                script_tags = f"\n<!-- Custom JS Scripts for {position} Start -->\n"
                for script_name, script_content in position_scripts:
                    script_tags += f"<script>/* {script_name} */\n{script_content}\n</script>\n"
                script_tags += f"<!-- Custom JS Scripts for {position} End -->\n"
                
                # 根据位置决定插入方式
                if position.startswith("after_"):
                    content_str = content_str.replace(tag, f"{tag}{script_tags}", 1)
                elif position.startswith("before_"):
                    content_str = content_str.replace(tag, f"{script_tags}{tag}", 1)

return content_str.encode('utf-8')
```

### 4.2 注入逻辑详解

1. **全局开关检查**：首先检查配置项 `SCRIPT_INJECTION_ENABLED`（默认为 `True`），只有启用时才执行注入

2. **遍历所有位置**：对 `script_manager.positions` 中的每个位置进行处理

3. **标签存在性检查**：只有当 HTML 内容中包含对应的标签（如 `<head>`、`</body>`）时才进行注入

4. **获取匹配脚本**：调用 `script_manager.get_scripts_for_position(position, domain)` 获取当前位置和域名下应注入的脚本

5. **构建 script 标签**：每个脚本被包装为内联 `<script>` 标签，格式如下：
   ```html
   <script>/* script_name.js */
   // 脚本内容
   </script>
   ```

6. **插入方式**：
   - `after_` 前缀的位置：将脚本插入到标签**之后**（如 `<head>` 后面）
   - `before_` 前缀的位置：将脚本插入到标签**之前**（如 `</body>` 前面）

7. **只替换第一个匹配**：使用 `replace(tag, ..., 1)` 确保只替换第一次出现的标签

### 4.3 生成的 HTML 示例

假设有以下配置：
- `dock.js` 和 `progress.js` 配置为 `after_head_start` 位置
- `target.js` 配置为 `before_body_end` 位置，且排除了 `example.com` 域名

当代理访问非 `example.com` 的网站时，原始 HTML：

```html
<html>
<head>
    <title>Example Page</title>
</head>
<body>
    <h1>Hello World</h1>
</body>
</html>
```

将被转换为：

```html
<html>
<head>
<!-- Custom JS Scripts for after_head_start Start -->
<script>/* dock.js */
// dock.js 的脚本内容
</script>
<script>/* progress.js */
// progress.js 的脚本内容
</script>
<!-- Custom JS Scripts for after_head_start End -->
    <title>Example Page</title>
</head>
<body>
    <h1>Hello World</h1>
<!-- Custom JS Scripts for before_body_end Start -->
<script>/* target.js */
// target.js 的脚本内容
</script>
<!-- Custom JS Scripts for before_body_end End -->
</body>
</html>
```

当代理访问 `example.com` 或 `google.com` 时，`target.js` 会被排除，只有 `dock.js` 和 `progress.js` 被注入。

## 7. 域名排除机制详解

域名排除使用**后缀匹配**方式，即 `domain.endswith(excluded)`：

```python
if domain and any(domain.endswith(excluded) for excluded in exclude_domains):
    continue
```

这意味着：
- 排除 `example.com` → 会排除 `example.com` 以及 `sub.example.com`、`www.example.com` 等子域名
- 排除 `google.com` → 会排除 `google.com`、`mail.google.com`、`www.google.com` 等

**注意**：这种后缀匹配可能导致意外排除。例如排除 `ample.com` 会同时排除 `example.com`，因为 `example.com` 以 `ample.com` 结尾。

## 8. 脚本缓存与热更新

### 8.1 缓存机制

`ScriptManager` 使用基于时间的缓存策略：

```python
self.scripts_cache = {}       # 缓存字典：{脚本名: 脚本内容}
self.last_load_time = 0       # 上次加载时间戳
self.cache_ttl = 60           # 缓存有效期（秒）
```

- 每次调用 `get_all_scripts()` 时，先检查缓存是否过期
- 未过期则直接返回缓存内容，避免频繁磁盘 I/O
- 过期则重新扫描 `scripts/` 目录，加载所有 `.js` 文件

### 8.2 配置热更新

脚本配置在 `ScriptManager.__init__()` 中加载，但**不会自动热更新**。如需更新配置，需要：
1. 修改 `databases/scripts.json`
2. 重启代理服务

但脚本**内容**本身支持热更新（60 秒缓存过期后自动重新加载）。

## 11. 设计特点与注意事项

### 11.1 设计特点

1. **内联注入**：脚本以 `<script>` 内联方式注入，而非 `<script src="...">` 外部引用，避免了额外的网络请求和跨域问题

2. **位置灵活**：支持四个注入位置，可以根据脚本功能选择最合适的注入点：
   - `after_head_start`：适合需要尽早执行的脚本（如 CSS 注入、全局变量定义）
   - `before_head_end`：适合需要在 head 中但依赖其他 head 资源的脚本
   - `after_body_start`：适合需要操作 DOM 但尽早执行的脚本
   - `before_body_end`：适合需要完整 DOM 树的脚本（最常用）

3. **域名排除**：可以对特定域名禁用脚本注入，避免在敏感网站（如银行、支付）上注入可能干扰功能的脚本

4. **独立开关**：每个脚本有独立的 `enabled` 字段，可以单独启用/禁用

5. **注释标记**：注入的脚本带有 HTML 注释标记，便于调试和识别

### 11.2 注意事项

1. **后缀匹配风险**：域名排除使用 `endswith()` 匹配，可能导致意外的域名排除（如第 7 节所述）

2. **缓存延迟**：脚本内容有 60 秒缓存，修改后不会立即生效

3. **仅限 HTML**：脚本注入仅在 `text/html` 类型的响应中执行

4. **首个标签匹配**：使用 `replace(tag, ..., 1)` 只替换第一个匹配的标签，如果 HTML 中有多个 `<head>` 或 `<body>` 标签，只有第一个会被处理

5. **配置不热更新**：`scripts.json` 的配置变更需要重启服务才能生效（但脚本内容可以热更新）

6. **无 CSP 处理**：注入逻辑没有处理目标站点的 Content-Security-Policy 头，如果目标站点设置了严格的 CSP，注入的内联脚本可能被浏览器阻止执行
