# SilkRoad-Next 产品需求文档 (PRD)

## 一、 产品概述
### 1.1 项目背景与目标
SilkRoad-Next 是一款高性能、可扩展的反向代理软件。其核心目标是帮助用户突破网络访问限制、保护隐私安全并提升连接速度。项目采用渐进式的架构设计，从基础的网页静态资源代理（V1），逐步演进到支持脚本注入、流媒体、WebSocket 乃至高级 WAF（Web应用防火墙）穿透的全能型代理网关。

### 1.2 目标用户与使用场景
- **目标用户**：需要突破网络访问限制的个人/企业、关注隐私保护的用户、网络安全研究人员与开发者。
- **核心场景**：
  - 隐藏真实 IP 访问受限网站，保护隐私。
  - 对目标网站内容进行实时修改、脚本注入与调试。
  - 在复杂网络环境下保持稳定、快速的网页及流媒体访问。

### 2.1 目录结构与模块划分

### 2.2 版本迭代路线图 (Roadmap)
- **MVP (V1) - 基础代理与重写核心**：
  构建系统的最小可行性版本。包含程序主入口 (`SilkRoad.py`)、配置加载 (`cfg.py`)、核心代理引擎 (`proxy.py`)。实现对 HTML/CSS/JS/XML/JSON 资源的完整解析与 URL 修正 (`url/`)。建立静态网站服务器 (`pageserver.py`, `pages/`)、日志系统 (`logging.py`, `logs/`)、UA 伪装 (`ua.py`)、控制台命令 (`command.py`) 与优雅退出机制 (`exit.py`)。
- **V2 - 性能优化、访问控制与定制化**：
  引入高并发支持与定制化能力。新增目标服务器连接池 (`connectionpool.py`) 与密集计算线程池 (`threadpool.py`) 以提升性能；加入会话管理 (`sessions.py`) 与缓存管理 (`cachemanager.py`) 优化加载速度；实现黑名单拦截 (`blacklist.py`) 与前端 JS 脚本注入 (`scripts.py`, `Scripts/`)，支持如浮动面板、进度条等增强功能。
- **V3 - 现代 Web 流支持**：
  增加对复杂流媒体与长连接的支持 (`stream/`)。涵盖媒体流 (`media.py`)、Server-Sent Events (`sse.py`) 等非传统 HTTP 请求的处理，优化大文件分块传输。
- **V4 - 全双工通信与流量控制**：
  全面支持 WebSocket 协议的代理与会话状态维护 (`websockets.py`)，并引入流量控制器 (`controler.py`)，实现请求调度与带宽管理。
- **V5 - 隐蔽与穿透**：
  引入 WAF 穿透模块 (`wafpasser.py`)，实现请求特征混淆、反爬虫机制绕过，极大提升代理在严格网络环境下的可用性。

## 三、 详细功能需求

### 3.1 核心反向代理与 URL 修正引擎 (V1 核心价值)
为了确保用户通过代理服务器访问目标网站时，页面内的所有跳转和资源加载都经过代理，必须实现极致的 URL 修正。
- **精准匹配与解析**：
  - 基于代码结构解析与多维正则表达式组合，精准定位不同类型文件中的 URL 资产。
  - **性能约束**：系统启动或模块加载时，所有正则表达式**必须进行预编译（Pre-compile）**，以应对高并发下的正则匹配性能损耗。
- **智能相对链接补全**：
  - 根据当前请求的 URL、Referer 或 Origin 自动补全相对链接。
  - **专属回溯算法**：内置路径状态机，确保在目标网站发生多次 301/302 重定向跳转后，仍能正确计算并补全绝对路径。
- **防乱码与编码兼容**：
  - 智能探测目标网站的字符集（Charset），在执行文本解析与正则替换前进行正确解码，并在返回给客户端前恢复原有编码。
- **处理链路编排**：
  1. 接收响应 -> 2. 根据 `Content-Encoding` 解压缩 -> 3. 补全相对 URL -> 4. 修正绝对 URL 为代理地址 -> 5. 使用 **gzip 或 deflate** 重新压缩 -> 6. 返回客户端。
- **服务端压缩限制**：
  - 代理服务器向目标网站发送请求时，`Accept-Encoding` 请求头仅声明支持 **gzip 和 deflate** 压缩方式。
  - 若目标网站返回其他压缩格式（如 br、zstd 等），代理服务器将无法正确解压，需按透明代理模式透传或返回错误提示。
- **URL 智能规范化（去重与补全）**：
  - 自动清理因重复代理导致的嵌套 URL。例如：
    - `.../127.0.0.1:8080/127.0.0.1:8080/www.baidu.com...` 修正为 `.../127.0.0.1:8080/www.baidu.com...`
    - `.../www.baidu.com/www.baidu.com...` 修正为 `.../www.baidu.com...`
  - 协议缺省补全：若目标 URL 未指定协议头，自动补全 `https://`。

### 3.2 性能与并发架构 (V2 核心升级)
- **全异步 I/O 模型**：采用统一的异步请求模式（协程机制），实现请求接收与目标服务器转发的并行处理，避免 I/O 阻塞。
- **连接与线程管理**：
  - `connectionpool.py`：维护与高频目标服务器的长连接（Keep-Alive），降低 TLS 握手开销。
  - `threadpool.py`：将 CPU 密集型任务（如解压缩、大规模正则替换）交由独立线程池处理，防止阻塞主异步事件循环。
- **统一压缩标准**：为优化客户端传输带宽，限定代理服务端向客户端下发的内容**统一使用 gzip 或 deflate 算法**进行压缩（需确保客户端兼容性或降级方案）。

### 3.3 内置控制台与命令管理 (V1)
提供一套 RESTful 风格或命令行的内置控制接口，访问 `/command` 路径可查看全量命令：
- `/command/start`：启动或恢复代理服务。
- `/command/pause`：暂停服务（拒绝新请求，但会平滑处理完当前队列中的请求）。
- `/command/exit`：触发优雅退出（Graceful Shutdown），保存配置与缓存后终止进程。
- `/command/status`：输出实时监控面板数据（含：QPS、并发连接数、内存占用、CPU负载、累计流量等）。
- `/command/clear`：一键清除本地服务端缓存，并向下游下发清除客户端（浏览器）缓存的指令。

### 3.4 静态网站服务器 (V1)
`pageserver.py` 是一个静态文件服务器，负责托管 `pages/` 目录中的静态网站资源。
- **技术栈无关**：前端可采用 TypeScript + 现代框架（React/Vue/原生TS）开发，构建产物放入 `pages/` 目录即可。
- **路由映射配置**：在 `databases/config.json` 中配置 URL 路径与 `pages/` 子目录的映射关系。
  ```json
  {
    "pageRoutes": {
      "/": "main",
      "/admin": "admin",
      "/error": "error"
    }
  }
  ```
  上述配置表示：
  - `/` 请求映射到 `pages/main/` 目录
  - `/admin/*` 请求映射到 `pages/admin/` 目录
  - `/error/*` 请求映射到 `pages/error/` 目录
- **核心功能**：
  - 静态文件托管（HTML、CSS、JS、图片等）
  - 自动识别 MIME 类型
  - 支持默认首页（index.html）
  - 路径映射与重写

### 3.5 日志与监控体系 (V1)
- **日志引擎**：强制基于 `loguru` 库实现，提供颜色高亮、多级别（INFO/DEBUG/ERROR/WARN）的日志记录。
- **存储与轮转**：
  - 统一存储至根目录 `logs` 文件夹。
  - 采用按天轮转（Daily Rotation），命名规范为 `年-月-日.log`（如：`2026-4-17.log`）。
- **生命周期管理**：
  - 日志自动清理策略可在 `databases/config.json` 中配置。
  - 支持的配置枚举：`7天` / `30天` / `6个月` / `1年` / `关闭`（不自动清理）。

## 四、 非功能需求

### 4.1 开发与工程规范
- **依赖管理**：所有第三方依赖及版本号必须明确记录在 `requirements.txt` 中。
- **代码规范**：核心算法（如 URL 补全、编码转换）必须包含完整的 Docstring 及边界测试用例。

### 4.2 性能与容量需求
- 单节点（标准 2C4G 配置）需支持至少 2000+ 并发长连接。
- URL 解析与重写引擎的单次处理延迟增加必须控制在 **50ms** 以内。

### 4.3 安全性需求
- 配置文件（`config.json`, `blacklist.json`）需支持热重载，避免重启服务导致连接中断。
- 提供 IP 访问频率限制（Rate Limiting），防止恶意客户端消耗代理服务器资源。

### 4.4 错误处理与降级机制
- 当目标网站无响应或返回 5xx 错误时，需向客户端返回友好的静态错误页面（来源于 `pages/error/` 目录），而非抛出代码异常。
- 当遇到无法解析的未知二进制流时，自动跳过 URL 修正步骤，直接以透明代理（Pass-through）模式透传数据。
