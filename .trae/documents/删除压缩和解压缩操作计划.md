# 删除压缩和解压缩操作计划

## 背景

用户指出 aiohttp 已经自带了压缩和解压缩功能，因此需要删除代码库中手动实现的压缩和解压缩操作。

## 当前状态分析

### 压缩/解压缩代码位置

主要在 `modules/flow/normal.py` 文件中：

1. **解压缩操作**：
   - `_decompress_content` 方法（第 543-570 行）：异步解压 gzip 和 deflate 格式
   - `_decompress_content_sync` 方法（第 572-596 行）：同步解压方法，用于线程池

2. **压缩操作**：
   - `_compress_content` 方法（第 598-616 行）：使用 gzip 压缩响应内容

3. **相关导入**：
   - `import gzip`（第 2 行）
   - `import zlib`（第 3 行）

4. **调用位置**：
   - `_forward_with_pool` 方法中调用 `_decompress_content_sync`（第 204-209 行）或 `_decompress_content`（第 211-214 行）
   - `_send_response` 方法中调用 `_decompress_content`（第 385-388 行）
   - `_send_cached_response` 方法中调用 `_compress_content`（第 514 行）
   - `_forward_with_pool` 方法中调用 `_compress_content`（第 255 行）
   - `_send_response` 方法中调用 `_compress_content`（第 403 行）

## aiohttp 自动处理机制

### aiohttp 自动解压

aiohttp 的 `ClientResponse.read()` 方法会**自动解压缩**响应内容：
- 当响应头包含 `Content-Encoding: gzip` 或 `Content-Encoding: deflate` 时
- aiohttp 会自动解压缩内容
- 返回的 `content` 已经是解压后的原始数据

### aiohttp 自动请求压缩头

aiohttp 会自动添加 `Accept-Encoding: gzip, deflate` 请求头，告知服务器客户端支持压缩。

## 实施计划

### 步骤 1：删除解压缩相关代码

1. 删除 `_decompress_content` 方法（第 543-570 行）
2. 删除 `_decompress_content_sync` 方法（第 572-596 行）
3. 删除 `_forward_with_pool` 中的解压缩调用（第 204-214 行）
4. 删除 `_send_response` 中的解压缩调用（第 385-388 行）

### 步骤 2：删除压缩相关代码

1. 删除 `_compress_content` 方法（第 598-616 行）
2. 删除 `_forward_with_pool` 中的压缩调用（第 255 行）
3. 删除 `_send_response` 中的压缩调用（第 403 行）
4. 删除 `_send_cached_response` 中的压缩调用（第 514 行）

### 步骤 3：调整响应头处理

1. 删除 `Content-Encoding` 头的设置逻辑（因为不再手动压缩）
2. 删除 `Content-Length` 头的重新计算逻辑（直接使用原始长度）
3. 保留 `Content-Encoding` 头的移除逻辑（因为 aiohttp 已解压，需要移除原始头）

### 步骤 4：清理导入

1. 删除 `import gzip`（第 2 行）
2. 删除 `import zlib`（第 3 行）

### 步骤 5：更新相关逻辑

1. 在 `_forward_with_pool` 中：
   - 直接使用 `await response.read()` 获取内容
   - 不再调用解压缩方法
   - 不再调用压缩方法
   - 直接使用原始内容长度设置 `Content-Length`

2. 在 `_send_response` 中：
   - 直接使用 `await response.read()` 获取内容
   - 不再调用解压缩方法
   - 不再调用压缩方法

3. 在 `_send_cached_response` 中：
   - 不再调用压缩方法
   - 直接使用缓存内容

### 步骤 6：更新配置

1. 删除或注释掉配置文件中与压缩相关的配置项：
   - `proxy.enableCompression`
   - `proxy.compressionLevel`

## 预期结果

1. 代码更简洁，删除约 100 行代码
2. 减少不必要的压缩/解压缩操作
3. 利用 aiohttp 的自动处理机制，提高性能
4. 避免重复解压缩（aiohttp 已解压，手动再解压会导致错误）

## 风险评估

### 低风险

- aiohttp 的自动解压是成熟稳定的功能
- 删除手动压缩/解压缩不会影响功能正确性

### 需要注意

1. **响应头处理**：确保正确移除 `Content-Encoding` 头，因为内容已被 aiohttp 解压
2. **Content-Length**：需要正确设置内容长度，因为解压后长度会变化
3. **缓存内容**：缓存的内容应该是解压后的原始内容

## 验证计划

1. 测试普通 HTTP 请求是否正常工作
2. 测试 gzip 压缩的响应是否正确处理
3. 测试 deflate 压缩的响应是否正确处理
4. 测试大文件传输是否正常
5. 测试缓存功能是否正常
